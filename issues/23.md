### 💡 Automatic job triggering — no manual steps needed (raw idea ⚠️)

Instead of requiring the admin to run CLI scripts or configure cron, the extension can drive its own job execution using MediaWiki's existing `Special:RunJobs` mechanism.

**How it would work:**
- After all Confluence migration jobs are enqueued, the special page fires an async HTTP POST to `Special:RunJobs` with `type=pandocConfluenceMigration` and a large `maxjobs` value — exactly what MW core does internally in `MediaWikiEntryPoint::triggerAsyncJobs()`
- Jobs start processing in the background immediately, no page views or cron required
- The Vue UI polls a status API endpoint for per-page progress
- On low-traffic wikis (our main audience), `$wgJobRunRate = 1` would mean jobs trickle through slowly on page views — self-triggering `Special:RunJobs` bypasses this entirely

**Fallback handling:**
- If `$wgRunJobsAsync = false` AND outbound HTTP is blocked: jobs fall back to running synchronously, one per page view. A warning can be surfaced in the UI if no progress is detected after N minutes
- If a proper cron/Redis job runner is set up: everything works even better automatically

**Why this beats TimedMediaHandler's approach:**
TimedMediaHandler excludes its jobs from the default queue and forces admins to run CLI commands manually. We avoid that entirely — migration starts from the UI and runs to completion without SSH access.

> ⚠️ **This idea is quite raw.** Needs validation across different hosting environments (shared hosts, locked-down setups, wikis with outbound HTTP restrictions). The `Special:RunJobs` self-triggering approach needs careful handling of the HMAC signature MW core uses to authenticate these requests.
